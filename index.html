Section 01

Purpose and Architectural Context

This specification defines the architecture, operator profile, and operational framework for the Superstructure Founding Validator Cohort on the Post Fiat (PFT) network.

The founding cohort operationalizes the institutional infrastructure delivered in the M2 Institutional Readiness layer. It establishes a set of validator operators capable of integrating with the Superstructure orchestration layer while preserving the zero-custody institutional deployment model established in M2.

This document defines five things:

  • Validator operator selection criteria and admission process
  • Minimum operational infrastructure standards with measurable thresholds
  • Integration boundaries between institutions, operators, and middleware
  • Custody and signing responsibilities under the zero-custody architecture
  • Onboarding procedures, operator lifecycle state transitions, and audit artifacts

The document is written to be readable by engineering teams, validator operators, and institutional risk and compliance stakeholders.

Three-Layer Architecture

The Superstructure architecture separates validator participation responsibilities across three distinct operational layers. No single layer can unilaterally execute validator control actions.

Layer 1

Institution

  • Holds custody of validator signing keys
  • Authorizes root-gated protocol actions
  • Participates in signing ceremonies
  • Controls governance vote execution
Layer 2

Validator Operator

  • Maintains validator node infrastructure
  • Manages network participation
  • Provides operational telemetry
  • Owns uptime and incident response
Layer 3

Superstructure Middleware

  • Monitors health signals across operators
  • Coordinates incident detection and escalation
  • Routes traffic and aggregates telemetry
  • Maintains audit log; holds no private keys
Section 02

Architectural Principles

All founding cohort operators must comply with these five non-negotiable principles. They are not guidelines. Deviations trigger a state transition to Degraded or Suspended.

Zero Custody Middleware

Superstructure middleware never holds validator private keys, signing material, or custody authority over digital assets at any point in the operational flow.

Institutional Key Authority

Validator signing keys remain inside the institutional custody environment at all times. Key material is never transmitted to operator or middleware infrastructure.

Operator Infrastructure Responsibility

Validator operators own infrastructure reliability, node configuration, and network participation performance. SLA obligations attach to the operator layer, not to middleware.

Deterministic Monitoring and Escalation

All validator health signals are observable by the Superstructure monitoring system. Observable telemetry is a contract requirement, not an optional integration feature.

Segregation of Duties

No single actor across the institution, operator, or middleware layer can unilaterally execute validator control actions. Every root-gated action requires cross-layer coordination.

Step 01

Operator Profile and Selection Criteria

The founding cohort must consist of Tier 1 validator operators that can support institutional infrastructure standards. The selection process is deterministic and evidence-based. Subjective assessments do not advance an operator to admission.

Eligible Operator Categories

  • Professional validator infrastructure providers — dedicated firms running validator infrastructure across multiple production networks
  • Institutional staking infrastructure firms — providers serving regulated financial institution clients with documented custody-compliant deployments
  • Enterprise cloud infrastructure operators — cloud-native operators running validator nodes on segregated enterprise infrastructure, not shared consumer tiers
  • Exchange-affiliated validator divisions — exchange-operated validator teams running fully segregated infrastructure that does not share resources with exchange operations
  • High-performance independent operators — independent operators with a verifiable, multi-network reliability record and documented operational procedures

Minimum Admission Criteria

An operator must meet all of the following before evaluation scoring begins. These are knock-out criteria — a single failure disqualifies the application.

  • Demonstrated validator infrastructure history on at least two production proof-of-stake networks
  • Dedicated operations personnel with verified capacity for 24-hour operational coverage
  • Written incident response procedures that specify detection time, escalation path, and resolution SLA
  • Documented infrastructure security hardening practices aligned to an accepted industry baseline (e.g., CIS Benchmarks)
  • Technical capability to provide signed operational telemetry to the Superstructure monitoring integration

Admission Artifacts

Operators must submit the following artifacts before evaluation scoring begins:

  1. Infrastructure architecture overview (logical topology, not physical addresses)
  2. Network topology summary showing peer connectivity design
  3. Monitoring and alerting procedures with detection time commitments
  4. Security hardening documentation referencing the specific baseline applied
  5. Operational incident escalation plan with named escalation paths
Step 02

Minimum Operational Standards

Operators in the founding cohort must maintain infrastructure that meets the following thresholds. These are measured values, not aspirational targets. Operators failing sustained threshold compliance trigger the lifecycle state machine described in Section 08.

Minimum Monthly Uptime
99.5%

Measured per calendar month. Downtime exceeding threshold triggers incident reporting.

Degradation Detection Window
≤ 5 min

Automated monitoring must detect validator degradation within five minutes of onset.

Minimum Geographic Regions
2+

Infrastructure deployed across at least two distinct geographic regions per operator.

Upstream Network Providers
≥ 2

Redundant upstream internet providers to prevent single-carrier outages.

Latency and Network Performance

Operators must maintain network connectivity sufficient to ensure stable ledger participation on the Post Fiat network. Specific requirements include:

  • Low propagation delay to validator peer nodes
  • Redundant upstream network providers (minimum two, from different transit providers)
  • Multiple active peer connections to the validator network at all times
  • Continuous peer connectivity monitoring with automated alerts on connection drops

Geographic Distribution

The founding cohort must collectively represent multiple geographic zones to minimize correlated outage risk from regional infrastructure failures, regulatory events, or natural disasters.

Preferred regional representation across the cohort:

  • North America (United States, Canada)
  • Europe (EU member states preferred for regulatory alignment)
  • Asia-Pacific (Singapore, Japan, or Australia preferred)

Operators must disclose infrastructure regions used for all active validator nodes. Changes to infrastructure region must be reported to Superstructure operations within 72 hours.

Infrastructure Redundancy

Each operator must maintain redundant infrastructure components. The specific requirements are:

  • Hot standby or warm backup node instances capable of resuming participation within the incident response window
  • Network path redundancy — no single network device whose failure takes the validator offline
  • Power redundancy at the facility level where data center colocation is used
  • Documented recovery runbooks for each class of infrastructure failure

Measurement Methodology

Uptime is measured from Superstructure middleware telemetry, not operator self-reporting. Telemetry data is the authoritative source for SLA compliance calculations. Operators may dispute readings by submitting infrastructure logs within 48 hours of a reported downtime event. Disputes are resolved by comparing third-party monitoring data against operator logs.

Step 03

Integration Boundaries

The Superstructure architecture defines strict operational boundaries between the institution, operator, and middleware layers. These boundaries are not preferences. They are enforced by technical controls and contractual obligation.

Responsibility Institution Operator Middleware
Hold validator signing keys ✓ Required ✗ Prohibited ✗ Prohibited
Operate validator node infrastructure ✗ No ✓ Required ✗ No
Aggregate health telemetry ✗ No ✗ No ✓ Required
Execute governance actions ✓ Required ✗ Prohibited ✗ Prohibited
Detect and escalate incidents ✗ No ✓ Self-report ✓ Independent
Participate in signing ceremony ✓ Required ✗ Excluded ✗ Excluded
Manage software version upgrades ✗ No ✓ Required ✗ No
Maintain audit log ✗ No ✗ No ✓ Required

Telemetry Data Flow

Operational telemetry flows in one direction: from validator operators to the middleware layer. The middleware layer does not push configuration or commands to operator nodes. The integration is read-only from the middleware perspective.

Required telemetry signals include:

  • Node availability status (up/down/degraded)
  • Consensus participation status (active/lagging/offline)
  • Peer connectivity count and latency
  • Validator performance signals (attestations, proposals, missed duties)
  • Software version currently running
Integration Contract Telemetry must be delivered via the Superstructure monitoring API using the signed telemetry schema defined in the M2 Integration Specification. Unsigned or schema-non-compliant telemetry is rejected and triggers a data-gap alert. Data gaps lasting more than 30 minutes constitute an operational incident.

What Middleware Cannot Do

The following actions are architecturally prohibited for the middleware layer. Operators may rely on these constraints as part of their security model:

  • Operate or access validator node infrastructure
  • Control validator software or configuration
  • Generate, store, or access private keys of any kind
  • Execute protocol governance actions
  • Transmit validator keys between layers
Step 04

Zero-Custody Operational Responsibilities

The founding validator cohort operates under strict custody boundaries defined in the M2 custody model. Zero-custody is not a feature — it is the foundational security property of the entire architecture. Every operational procedure must preserve this property.

Validator Key Generation

Validator signing keys must be generated entirely within the institutional custody environment. Key generation outside of approved custody infrastructure is a critical compliance violation that requires immediate remediation and incident disclosure.

Acceptable key generation environments:

  • Hardware Security Modules (HSMs) — FIPS 140-2 Level 3 or equivalent, operated by the institution
  • Air-gapped signing systems — physically isolated systems with documented access controls and chain-of-custody logs
  • Institutional multisignature custody environments — institutional-grade multisig platforms (e.g., Fireblocks, BitGo) operating under the institution's custody agreement
Prohibited Key Environments Validator keys must never be generated on operator infrastructure, cloud instances accessible to operators, middleware servers, or any environment outside institutional custody control. Software wallets on internet-connected machines are not acceptable for founding cohort validators.

Signing Ceremony Protocol

Root-gated protocol actions require execution of the signing ceremony workflow defined in the M2 Signing Ceremony specification. The workflow enforces zero-custody guarantees at every step.

  1. Payload Construction

    Superstructure middleware constructs the unsigned transaction payload and delivers it to the institution for review. Middleware never holds or generates key material at this step.

  2. Institutional Verification

    Institution verifies the payload parameters — validator address, action type, parameters — against expected values before authorizing signature. This step is mandatory and must be performed by an authorized institution representative, not an automated process.

  3. Signing Inside Custody Environment

    Institution executes the signing operation inside the custody environment. Private key material never leaves the custody boundary. The output is a signed transaction, not key material.

  4. Signed Transaction Relay

    The signed transaction is relayed to the network for broadcast. At no point does middleware or operator infrastructure gain access to the private key used to produce the signature.

Operational Separation Summary

Core Invariant Institution controls keys. Operator runs infrastructure. Middleware coordinates monitoring and routing. These roles must never overlap. If any of these three statements becomes false, the zero-custody architecture is compromised.
Section 07

Operator Lifecycle and State Machine

Validator operators move through defined lifecycle states. State transitions are triggered by operational metrics, incident history, or governance decisions — not discretionary judgment calls by Superstructure operations staff.

Candidate

Operator undergoing evaluation against scoring framework.

→
Approved

Accepted into founding cohort. Onboarding begins.

→
Active

Running validator infrastructure. Telemetry flowing. SLA measured.

Active

Normal operating state.

→
Degraded

Operational issues detected. Remediation window open.

→
Suspended

Temporarily removed from routing. Root-cause analysis required.

→
Removed

Permanently removed. Cannot re-apply within 12 months.

Transition Triggers

  • Active → Degraded: Monthly uptime below 99.5%, detection time exceeding 5 minutes for three or more incidents in a 30-day window, or telemetry data gaps exceeding 30 minutes
  • Degraded → Active: Operator demonstrates remediation and maintains full compliance for 14 consecutive days
  • Degraded → Suspended: Failure to remediate within the remediation window (14 days), or a second critical incident during Degraded status
  • Suspended → Active: Requires root-cause analysis submission, governance review, and 30-day probationary monitoring period
  • Suspended → Removed: Second suspension within 12 months, or failure to submit root-cause analysis within the required window
Section 08

Operator Evaluation Scoring Framework

Operators are evaluated using a deterministic scoring framework. Scores are derived from submitted artifacts and verifiable operational history. Self-attestation without supporting evidence does not score.

Category Weight What Is Measured Evidence Required
Operational Reliability 30% Historical validator uptime across two or more production networks over the prior 12 months Third-party monitoring data or signed attestation from a prior network foundation
Infrastructure Resilience 25% Geographic redundancy, failover design, backup instance availability Architecture diagram, colocation agreements, failover runbook
Operational Transparency 20% Ability to provide telemetry, reporting cadence, incident disclosure history Sample telemetry output, prior incident disclosures if applicable
Security Posture 15% Infrastructure hardening baseline applied, access control model, key management practices Security hardening documentation referencing CIS Benchmarks or equivalent
Institutional Readiness 10% Capacity to participate in structured escalation, operational reporting, and governance coordination Named escalation contacts, written escalation procedure, prior institutional client references if available
Admission Threshold Operators must achieve a weighted score of 75 or above to be admitted to the founding cohort. Operators scoring 65–74 may be admitted on a conditional basis with a defined 60-day improvement plan. Operators scoring below 65 are not admitted and may reapply in 6 months with new evidence.
Section 09

Incident Coordination Model

Operational incidents follow escalation workflows defined in the M2 SLA framework. Two detection mechanisms operate in parallel — operator self-reporting and independent middleware monitoring. Both must function correctly. A failure of either is itself an operational deficiency.

When a Degradation Event Occurs

  1. An incident record is generated in the Superstructure audit log with a timestamp and event classification
  2. The operator is notified via the registered escalation channel within 10 minutes of detection
  3. If root-gated remediation is required, the institution receives a signed notification payload
  4. The incident remains open in the audit log until the operator submits a resolution confirmation and post-incident summary

Operator Incident Reporting Obligations

  • Operators must self-report infrastructure incidents via the Superstructure operations channel within 15 minutes of detection
  • Self-reports must include: incident type, estimated impact, current status, and estimated resolution time
  • Post-incident summaries are due within 48 hours of resolution for incidents exceeding 30 minutes of downtime

All incident events, including self-reports, middleware detections, resolution confirmations, and post-incident summaries, are recorded in the Superstructure audit log and available to institutions during due diligence.

Section 10

Governance Coordination

The founding validator cohort participates in Post Fiat network governance coordination activities. This does not mean operators control governance outcomes. It means operators have defined responsibilities for staying informed, coordinating upgrade timelines, and surfacing operational constraints to the governance process.

Operator Governance Responsibilities

  • Monitor protocol upgrade proposals on the cadence set by the Post Fiat governance process
  • Provide operational impact assessments for proposed changes that affect validator infrastructure within the comment period
  • Coordinate network upgrade deployment timelines with Superstructure operations before execution
  • Maintain active communication channels with Superstructure operations for governance coordination
Key Authority Boundary Governance actions requiring validator key authorization remain under institutional control. Operators do not hold keys and cannot execute on-chain governance actions. Operator governance participation is advisory and coordination-focused.

This specification was designed to align with the institutional trust infrastructure delivered in the M2 phase. The following table maps M3 operational requirements to the corresponding M2 components.

M2 Component M3 Alignment
Zero-Custody Model Operators never hold validator keys. Key generation and storage remain entirely within institutional custody environments. Middleware is architecturally excluded from key access.
Signing Ceremony Specification Institutions authorize all root-gated actions using the four-step signing ceremony. No root-gated action executes without institutional signature inside the custody environment.
SLA Framework Operators provide signed telemetry signals enabling monitoring and escalation. Middleware uses telemetry as the authoritative source for SLA compliance measurement.
Disaster Recovery and Business Continuity Operators maintain geographic redundancy, backup instances, and documented recovery runbooks supporting validator availability requirements.

Regulatory Compatibility Mapping Institutional key authority and operator infrastructure separation align with the custody-compliance frameworks mapped in M2 for regulated institutional participants.
Section 12

Audit Evidence Artifacts

Validator operators participating in the founding cohort must produce the following artifacts for institutional due diligence. These artifacts allow institutions to independently evaluate operator readiness before committing validator deployment to a specific operator.

  • Infrastructure architecture documentation — logical topology showing node configuration, redundancy design, and network connectivity model
  • Operational monitoring procedures — documentation of automated monitoring tools, alert thresholds, and on-call coverage model
  • Incident response procedures — written runbooks covering at minimum: node failure, network partition, and software upgrade failure scenarios
  • Security hardening documentation — reference to the specific hardening baseline applied (CIS Benchmarks, NIST 800-53, or equivalent), plus evidence of implementation
  • Operational uptime reporting — 12 months of uptime history from a third-party monitoring source, or equivalent attestation for operators without 12-month history on any network
  • Geographic infrastructure disclosure — list of regions where validator nodes are deployed, updated whenever changes occur
  • Telemetry integration test results — evidence of successful telemetry delivery to the Superstructure monitoring API prior to activation

All artifacts are retained in the Superstructure operator registry and made available to participating institutions upon request. Artifacts are updated on a 12-month review cycle or on material change, whichever occurs first.

Section 13

Cohort Formation Strategy

The founding cohort begins with a limited number of operators. A smaller initial cohort allows Superstructure operations to validate the monitoring integration, incident coordination model, and escalation workflows before scaling to a larger operator set.

Recommended Initial Cohort Size Five to ten operators. This range provides sufficient geographic diversity and infrastructure redundancy across the cohort while remaining operationally manageable during the initial activation phase.

Cohort Composition Principles

  • Operators should represent diverse infrastructure environments (cloud providers, data center colocation, geographic regions) to minimize correlated failure risk
  • At least one operator should be present in each of the three preferred geographic zones: North America, Europe, and Asia-Pacific
  • No single infrastructure provider should represent more than 40% of cohort validators to limit platform concentration risk
  • Cohort composition is reviewed after the first 90 days of operation and adjusted based on operational performance data

Expansion Path

As operational experience increases and monitoring integration matures, the cohort will expand. New operator applications are processed against the same scoring framework as founding cohort operators. There is no grandfathering of standards for later cohort members. Cohort expansion proposals require Superstructure governance review.
